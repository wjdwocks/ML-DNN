### MMVQShape
- learning rate = 1e-4를 그대로 fix하는 것으로.
    - lr = 1e-2, 1e-3, 5e-4, 2e-4 까지 다 해봤는데, 1e-4에서 codebook 모양새나, Acc가 다 좋았음.
- lambda_s, lambda_z, lambda_x, lambda_dist 를 최대한 좋은 것을 맞추고, 다른 것을 만져볼 예정.
    - lambda_s = 3, lambda_z = 2 에서 일단 다양한 codebook이 추출되었다. lambda_x, lambda_dist는 아직..
- 가장 잘 되었던 lambda 조합 후보들에 대해서, codebook을 늘려보기도 하고, token_length도 늘려보며 더 실험해보자.
- 근데, codebook이 잘 나오면 Acc가 줄어드는 경우가 많은데, 이것을 어떻게 해야할 지 고민중.
    - 이게 lambda_s가 높은 값에서 발산하는 경향때문에 그런 것 같기도 해서 엄청 줄여서도 wandb에서 확인을 해봐야 함.

lambda_s = 3, lambda_z = 2 에서 우선 가장 좋았음. 
lambda_s = 5, lambda_z = 3 에서도 나쁘진 않았음. (lambda_s = 5, lambda_z = 1에선 별로), (lambda_s = 3, lambda_z = 3에서도 별로.)
→ 근데 이 때 문제는 lambda_x는 1로 고정했을 때라서, Recon을 한 후 Acc가 아주 낮아진다는 문제가 있음.
→ 그래서 lambda_s=3, lambda_z=2를 고정하고, lambda_x를 찾아보려고 했는데, lambda_x = 5, 3, 2 에서 다 별로(x의 비중이 커지면 Acc는 높아지지만, codebook이 단조로워짐.)
→ 그래서 lambda_s=5, lambda_z=3에서 lambda_x를 2로 두고(안되면 1.5 1.2 이렇게도 해봐야함.) 결과를 보자.

→ Acc보다 Codebook에 집중해서 lambda_x = 1, lambda_z = 3, lambda_s = 5로 고정하고 다른것들을 확인한 결과, codebook = 32인 경우에도 상당히 잘 뽑힌 것을 볼 수 있음.
